
# ADR-001: Uso de SQLite como base de datos inicial

## Contexto
Beholder necesita una base de datos para almacenar clientes, conexiones, planes, nodos y suscriptores (fibra y antena).  
El sistema está en fase inicial, con un equipo reducido y foco en rapidez de desarrollo y despliegue.  
Las opciones consideradas fueron:
- **PostgreSQL**: robusto, escalable, multiusuario, con soporte avanzado de índices y particiones.  
- **SQLite**: liviano, embebido, sin necesidad de servidor, fácil de administrar.  

El entorno actual es un ISP en crecimiento, con despliegues rápidos y necesidad de simplicidad operativa.

## Decisión
Se decidió usar **SQLite** como base de datos inicial para Beholder.  
Motivos:
- Simplicidad de instalación y cero mantenimiento de servidor.  
- Integración directa con Python sin dependencias externas.  
- Suficiente rendimiento para el volumen actual de clientes y antenas.  
- Facilita pruebas y despliegues rápidos en entornos de desarrollo y producción.  

## Consecuencias
**Positivas:**
- Menor complejidad operativa.  
- Deploy más sencillo (solo archivo `.db`).  
- Ideal para prototipado y validación rápida.  

**Negativas:**
- Limitaciones de concurrencia (solo un escritor a la vez).  
- Escalabilidad reducida para grandes volúmenes de datos.  
- Migración futura necesaria a PostgreSQL u otro motor más robusto.  

**Mitigación:**
- Diseño modular de acceso a datos (`db.py`) para facilitar migración.  
- Documentación clara del esquema y relaciones.  
- Plan de roadmap para migrar a PostgreSQL cuando el volumen lo requiera.
